# Data Model Specification

**Authors:** Syed Ali Turab, Fergie Feng & Diana Liu | **Team:** Broadview | **Date:** March 1, 2026

---

## 1. Purpose

This document defines the data model for the PetCare Triage & Smart Booking Agent.

The model supports:

- Rule-grounded triage and routing
- Scheduling and appointment booking
- Structured intake handoff to veterinary staff
- Safety-first operations with minimal PII storage

For the **POC**, all data lives in static JSON files under `backend/data/`. The schema below defines the logical structure so that a production migration to PostgreSQL, Firebase, or any relational store can happen without redesigning agent logic.

---

## 2. Data Domains

### 2.1 Clinic Configuration (`clinic_rules.json`)

Stores clinic-approved logic and templates used by safety-constrained agents.

- Triage rules: urgency thresholds, trigger keywords, and corresponding actions
- Symptom category mappings: routing from symptom domain to appointment type
- Provider list: veterinarian names, specialties, and availability pools
- Species-specific notes: behavioral differences between dogs, cats, exotics

### 2.2 Emergency Red Flags (`red_flags.json`)

A curated, conservative list of 50+ symptom phrases that trigger immediate escalation.

- Maintained separately from `clinic_rules.json` to allow independent updates
- Safety Gate Agent (B) matches owner input against this list
- When in doubt, the red flag is included (false positive > false negative)

### 2.3 Scheduling (`available_slots.json`)

Mock appointment availability for the demo clinic.

- Provider schedules with 30-minute slot granularity
- Slot status: `available` or `booked` (boolean `available` field)
- Clinic operating hours: weekday (9-5), Saturday (9-1), Sunday closed
- Scheduling Agent (F) reads this at runtime to propose appointment options

### 2.3b Illness Knowledge Base (`pet_illness_kb.json`) [Added v1.1]

Curated clinical reference data used by the Triage Agent's RAG retrieval layer to ground LLM urgency classification decisions.

- **24 illness entries** sourced from ASPCA, AVMA, Cornell Feline Health Center, and VCA
- Read by `backend/utils/rag_retriever.py` via `retrieve_illness_context()` — **never read directly by agents**
- Loaded once at first retrieval call via `@lru_cache(maxsize=1)` — zero reload overhead
- Categories: GI, respiratory, urological, toxicological, neurological, cardiovascular, dermatological, dental, reproductive, trauma/pain, parasitic, orthopedic, endocrine, ophthalmic, multi-system
- Top-3 matching entries are injected as `=== CLINICAL REFERENCE ===` block into the Triage Agent LLM system prompt
- This file is maintained separately from `clinic_rules.json` to support independent clinical knowledge updates without touching routing or scheduling logic

See Section 3.5b below for the full field schema.

### 2.4 Two-Tier Server-Side Session Store (In-Memory)

The POC uses a **two-tier in-memory session store** managed by the Orchestrator:

| Tier | Dict Key | TTL | Purpose |
|------|----------|-----|---------|
| **Active sessions** | `sessions` | 1 hour | In-progress intake conversations; full read/write by agents |
| **Completed sessions** | `completed_sessions` | 24 hours | Finished intakes; read-only for PDF download and summary retrieval |

- When an intake completes (state → `complete` or `emergency`), the session is moved from active to completed.
- A **background cleanup timer** runs every 10 minutes, evicting expired sessions from both tiers based on TTL.
- Sessions are lost on server restart (no persistence in POC); in production these would be backed by Redis or a database.

### 2.5 Appointments (In-Memory)

Booking confirmations generated by the Scheduling Agent. In the POC these exist only in session memory. In production, they would be written to a persistent store.

### 2.6 Client-Side Storage (Browser localStorage)

The frontend persists lightweight user data in the browser via `localStorage`:

| Key | Data | Purpose |
|-----|------|---------|
| `petcare_pet_profile` | Species, name, breed, age | Pre-fills intake for returning users |
| `petcare_symptom_history` | Array of past visits (date, complaint, urgency) | Recurring symptom awareness; displayed in symptom history panel |
| `petcare_dark_mode` | Boolean | User dark mode preference |
| `petcare_language` | Language code (e.g., `"fr"`) | Persisted language preference |

This data never leaves the browser and is not sent to the server unless the user explicitly submits it during intake.

## 3. MVP Data Schemas

### 3.1 `clinic_rules.json`

**File:** `backend/data/clinic_rules.json`
**Read by:** Routing Agent (E)
**Written by:** No agent (clinic-managed, static in POC)

| Field | Type | Description |
|-------|------|-------------|
| `version` | string | Schema version (e.g., `"1.0.0"`) |
| `clinic_name` | string | Demo clinic identifier |
| `triage_rules` | object | Keyed by urgency tier (`emergency`, `same_day`, `soon`, `routine`) |
| `triage_rules.{tier}.description` | string | Human-readable tier description |
| `triage_rules.{tier}.triggers` | string[] | Symptom keywords that map to this tier |
| `triage_rules.{tier}.action` | string | Recommended booking action |
| `routing_map` | object | Keyed by symptom category (e.g., `gastrointestinal`, `respiratory`) |
| `routing_map.{category}.appointment_type` | string | Appointment type code |
| `routing_map.{category}.providers` | string[] | Eligible provider names |
| `routing_map.{category}.notes` | string | Clinical notes for the category |
| `providers` | array | List of provider objects |
| `providers[].name` | string | Provider display name (e.g., `"Dr. Chen"`) |
| `providers[].specialties` | string[] | Clinical specialties |
| `species_notes` | object | Keyed by species; species-specific intake guidance |

---

### 3.2 `red_flags.json`

**File:** `backend/data/red_flags.json`
**Read by:** Safety Gate Agent (B)
**Written by:** No agent (curated list, static in POC)

| Field | Type | Description |
|-------|------|-------------|
| `version` | string | Schema version |
| `description` | string | Purpose of this file |
| `red_flags` | string[] | 50+ emergency symptom phrases for substring matching |

Current red-flag categories:
- Respiratory: breathing difficulty, gasping, blue/pale gums
- Bleeding: uncontrolled/heavy bleeding
- Neurological: seizures, convulsions, tremors, collapse, unresponsive
- Toxin ingestion: chocolate, antifreeze, rat poison, xylitol, grapes, raisins, medication
- Urinary: inability to urinate, blockage
- GI emergency: bloat, gastric dilation, non-productive retching
- Trauma: hit by car, fall from height, severe burns
- Other: heat stroke, hypothermia, snake bite, anaphylaxis, paralysis

---

### 3.3 `available_slots.json`

**File:** `backend/data/available_slots.json`
**Read by:** Scheduling Agent (F)
**Written by:** No agent in POC (Booking Agent would write in production)

| Field | Type | Description |
|-------|------|-------------|
| `version` | string | Schema version |
| `clinic_hours` | object | Operating hours per day type |
| `clinic_hours.weekday` | object | `{ "open": "09:00", "close": "17:00" }` |
| `clinic_hours.saturday` | object | `{ "open": "09:00", "close": "13:00" }` |
| `clinic_hours.sunday` | string | `"closed"` |
| `slot_duration_minutes` | integer | Default slot length (30) |
| `slots` | array | List of slot objects |
| `slots[].datetime` | string (ISO 8601) | Slot start time |
| `slots[].provider` | string | Provider name |
| `slots[].type` | string | Slot type (`general`, `dental`, `surgery`) |
| `slots[].available` | boolean | Whether the slot is open for booking |

### 3.5b Illness Knowledge Base Entry (`pet_illness_kb.json`) [Added v1.1]

**File:** `backend/data/pet_illness_kb.json`
**Read by:** `backend/utils/rag_retriever.py` (via `retrieve_illness_context()`) → Triage Agent (D)
**Written by:** No agent (manually curated from ASPCA/AVMA/Cornell/VCA, static in POC)

Top-level structure: `{ "version": "1.0", "description": "...", "illnesses": [ ...entries ] }`

| Field | Type | Description |
|-------|------|-------------|
| `id` | string | Unique identifier (e.g., `"GI-001"`, `"URIN-001"`) |
| `name` | string | Illness name (used in `format_rag_context()` output, not shown to owner) |
| `category` | string | Clinical category (e.g., `"gastrointestinal"`, `"urological"`) — used for category bonus scoring |
| `species` | string[] | Species this entry applies to (e.g., `["dog", "cat"]`) — used for species bonus (+2) |
| `keywords` | string[] | Keyword phrases for overlap scoring (e.g., `["vomiting", "not eating", "blood in stool"]`) |
| `typical_urgency` | string | Default urgency tier: `Emergency` / `Same-day` / `Soon` / `Routine` |
| `urgency_escalators` | string[] | Conditions that push urgency to a higher tier |
| `urgency_de_escalators` | string[] | Conditions that allow a lower tier classification |
| `key_triage_notes` | string | Clinical guidance injected into triage LLM system prompt |
| `red_flags` | string[] | Presentation signs in this category that require immediate escalation |
| `species_notes` | object | Keyed by species (`"dog"`, `"cat"`, etc.); species-specific clinical caveats |

**Key entries:**
- `URIN-001` — Urinary blockage (Emergency): fixes TC-04; "male cat straining with no output" is listed as an urgency escalator
- `GI-002` — GDV/Bloat (Emergency): "non-productive retching + distended abdomen" → immediate ER
- `TOXI-001` to `TOXI-004` — Toxin ingestion categories (Emergency): chocolate, xylitol, antifreeze, NSAIDs
- `NEURO-001` — Seizures/neurological (Emergency / Same-day depending on presentation)
- `REPRO-001` — Pyometra (Emergency for intact females with systemic signs)
- `ORTHO-001` — IVDD/spinal pain (Same-day to Emergency based on paralysis signs)

## 5. Data Access Policy

Role-based access control enforces minimal privilege:

| Agent | `clinic_rules` | `red_flags` | `available_slots` | `pet_illness_kb` | `intake_records` | `appointments` |
|-------|:-:|:-:|:-:|:-:|:-:|:-:|
| **Intake (A)** | -- | -- | -- | -- | -- | -- |
| **Safety Gate (B)** | -- | Read | -- | -- | -- | -- |
| **Confidence Gate (C)** | -- | -- | -- | -- | -- | -- |
| **Triage (D)** | -- | -- | -- | Read (via RAG retriever) | -- | -- |
| **Routing (E)** | Read | -- | -- | -- | -- | -- |
| **Scheduling (F)** | -- | -- | Read | -- | -- | Write |
| **Guidance & Summary (G)** | -- | -- | -- | -- | Write | -- |
| **Orchestrator** | Read | Read | Read | -- | Read/Write | Read |

- Only the Scheduling Agent writes appointment records
- Only the Guidance & Summary Agent writes intake records
- The Orchestrator has broad read access for coordination but delegates writes to the responsible agent
- No agent stores raw conversation transcripts in the POC

## 6. Privacy & Security

### 6.1 Data Minimization
- No full raw conversation transcripts are stored
- Only structured fields are captured
- Voice audio is transcribed and discarded (never persisted)

### 6.2 PII Separation
- Owner contact info is not collected in the POC
- In production, contact fields should be in a separate encrypted store
- Pet names and species are considered low-sensitivity

### 6.3 Retention / TTL
- POC: two-tier session TTLs — active sessions expire after **1 hour**, completed sessions after **24 hours**
- Background cleanup timer evicts expired sessions every **10 minutes**
- All in-memory sessions are lost on server restart
- Client-side localStorage data persists indefinitely in the browser (user can clear manually)
- Production: intake logs should have a configurable retention policy (e.g., 90 days)
- Webhook payloads should be anonymized or purged after processing

### 6.4 Synthetic Data
- All test scenarios use synthetic data
- No real patient or owner data is used in any demo or test

## 7. Data Flow Summary

```
1. Owner submits symptom description (free text)
2. Intake Agent (A) → produces structured pet profile + symptoms (in-memory)
3. Safety Gate (B) → reads red_flags.json, checks for emergency triggers
4. Confidence Gate (C) → validates required fields, scores confidence
5. Triage Agent (D) → RAG retriever scores complaint against pet_illness_kb.json → top-3 entries injected into LLM system prompt → classifies urgency tier via LLM (rule-based fallback if LLM fails)
6. Routing Agent (E) → reads clinic_rules.json routing_map, selects service line
7. Scheduling Agent (F) → reads available_slots.json, proposes slots
8. Guidance & Summary (G) → assembles owner guidance + clinic JSON summary
9. Orchestrator → returns response to frontend + POSTs to webhook (if configured)
```
